- **[User Preferences]**: Do not record low-level implementation details or one-time fixes in learnings. Focus on architectural decisions, package ownership, and explicit user preference overrides â€” entries that help any future agent on any task, not just the current one.
- **[Architecture]**: Sandboxed execution is based on provider-specific Docker sandboxes (`locus sandbox`, then `locus sandbox claude|codex` for auth), while model switching still happens through `ai.model`. Avoid docs that imply Docker install alone completes sandbox setup.
- **[Packages]**: CLI tests currently live in `packages/cli/__tests__`; verify script globs in `packages/cli/package.json` against this layout before relying on `test:unit`/`test:integration` commands.
- **[Packages]**: Never use `workspace:*` in package dependencies that get published to npm. `npm publish` does not resolve the `workspace:` protocol (unlike `pnpm publish`), so consumers get `EUNSUPPORTEDPROTOCOL` errors on install. Always use real semver versions (e.g., `"^0.21.7"`) for inter-package deps, even in `devDependencies`.
- **[Conventions]**: All packages are `@locusai/locus-<name>` on npm â€” no legacy unscoped `locus-<name>` format. Users reference them by short name (e.g., `locus pkg telegram`). `normalizePackageName` only handles short name â†’ `@locusai/locus-<name>` and pass-through of already-scoped names. For scoped npm packages, `.bin/` entries use the name after the scope (e.g., `locus-telegram` not `@locusai/locus-telegram`).
- **[Packages]**: Every publishable package MUST have a `"files"` field in `package.json` listing `"bin"` and `"dist"` (and any other runtime artifacts). Without it, npm falls back to the root `.gitignore` which excludes `dist` â€” causing published packages to ship without compiled code or binaries. See `packages/cli/package.json` and `packages/sdk/package.json` for reference.
- **[Packages]**: Every Locus package MUST include `"locusai-package"` in the `"keywords"` array of `package.json`. The web app marketplace (`apps/www/src/app/packages/page.tsx`) discovers packages via npm search with `keywords:locusai-package`. Without this keyword, the package won't appear on the packages page.
- **[Architecture]**: The SDK's `invokeLocusStream` uses `stdin: "ignore"` by design â€” it's for non-interactive streaming output. CLI commands that can enter interactive mode (exec REPL, discuss Q&A) must guard with `process.stdin.isTTY` checks. The telegram bot must validate that interactive commands (exec, discuss) receive arguments to prevent spawning interactive child processes.
- **[Architecture]**: The CLI is bundled via `bun build` into a single JS file â€” avoid native Node.js addons. For external tool integration (sox, whisper.cpp), use `child_process.spawn` instead.
- **[Architecture]**: Homebrew's `whisper-cpp` formula installs the binary as `whisper-cli` (not `whisper-cpp`). Voice input detection in `packages/cli/src/repl/voice.ts` checks multiple candidate names with `whisper-cli` first.
- **[User Preferences]**: Terminal keyboard shortcuts like Ctrl+Space and Shift+Space (Kitty protocol) are unreliable across terminal emulators. User prefers slash commands (`/voice`) over keyboard shortcuts for non-standard input modes.
- **[Architecture]**: Docker sandbox on WSL/Windows may mount the project at a different path than the host. `SandboxConfig.containerWorkdir` stores the detected container path (probed via `detectContainerWorkdir` after sandbox creation). All `-w` flags in `docker sandbox exec` and `enforceSandboxIgnore` cleanup scripts use `containerWorkdir ?? hostPath`. On macOS/Linux, paths match and `containerWorkdir` is null.
- **[Debugging]**: `docker sandbox exec <name> pwd` returns the container image's default WORKDIR, NOT the project mount point. Never rely on `pwd` for mount detection. On WSL, `docker sandbox ls` shows the workspace as a Windows UNC path (`\\wsl.localhost\...`), and the actual container mount point can be deeply nested (exceeding `find -maxdepth 4`). Use `/proc/mounts` parsing as primary strategy, deep `find -maxdepth 10` as fallback.
- **[Architecture]**: ENOSYS in Docker sandboxes has two root causes: (1) symlink creation fails on bind mounts (WSL/NTFS, restrictive configs) â€” fixed by installing on native fs then `cp -rL` back, and (2) seccomp blocks syscalls like `statx` â€” fixed by using `--privileged` flag on `docker sandbox exec`. Always use `--privileged` for all `docker sandbox exec` commands. Symlink support is probed at creation time via `probeSymlinkSupport()` and cached in `SandboxConfig.noSymlinks`.
- **[Architecture]**: `docker sandbox exec` supports `--privileged` flag (per Docker Desktop docs) which bypasses seccomp restrictions. It does NOT support `--security-opt`. Always use `--privileged` to avoid `statx`/seccomp ENOSYS errors with Zig/Bun/Biome binaries.
- **[Debugging]**: In `detectContainerWorkdir`, mount table candidate validation must check for actual project markers (`.git` dir or `package.json`), NOT merely `test -d` on the mount point itself. Docker internal mounts like `/run/secrets` are always directories and will produce false positives. Also filter `/run` as a system prefix alongside `/proc`, `/sys`, `/dev`.
- **[Architecture]**: CLI args are normalized via `normalizeArgs()` in `cli.ts` â€” single-dash long flags (e.g., `-sprint`) are converted to double-dash (`--sprint`). This is required because Telegram converts `--` to `-`. The normalization runs once at the top of `parseArgs` and covers all command-specific flags since they flow through as positional args.
- **[Architecture]**: Run state is stored per-sprint in `.locus/run-state/<sprint-slug>.json` (parallel runs use `_parallel.json`). Each sprint has independent state, so `sprint active` does NOT clear any run state. `saveRunState` derives the file path from `state.sprint`. The `run-state` directory is created during `locus init`.
- **[Architecture]**: `locus init` reinit preserves all top-level config sections (`ai`, `agent`, `sprint`, `logging`, `sandbox`, `packages`). The `packages` field holds community package configs (e.g., telegram bot tokens) â€” always ensure new config sections are included in the reinit merge logic in `init.ts`.
- **[Architecture]**: `@locusai/locus-gateway` is the channel-agnostic message gateway. It owns shared logic: `CommandTracker` (string session IDs), `CommandRouter` (parsing), `CommandExecutor` (CLI invocation + streaming), `COMMAND_REGISTRY` (commandâ†’CLI arg mapping), and `MessageFormatter`. Platform packages (telegram, discord, etc.) are thin adapters that depend on gateway. Build order: sdk â†’ gateway â†’ telegram â†’ cli.
- **[Conventions]**: New library packages (like gateway) follow the SDK pattern: dual ESM/CJS builds via `bun build`, declarations via `tsc -p tsconfig.build.json --emitDeclarationOnly`, `tsconfig.json` extends `../../tsconfig.base.json`, `tsconfig.build.json` overrides module to CommonJS for declaration generation. See `packages/sdk/` as the canonical template.
- **[Architecture]**: Sandbox shell (`handleShell`) must be language-agnostic â€” auto-detect common bin dirs for all ecosystems (node_modules/.bin, .venv/bin, $HOME/.cargo/bin, $HOME/go/bin, etc.) instead of hardcoding JS-only paths. Users can extend via `.locus/sandbox-profile.sh` for custom env vars and PATH entries.
- **[Architecture]**: `@locusai/locus-pm2` is the shared PM2 process management package. All PM2 operations take a `Pm2Config` (processName, scriptPath, scriptArgs) so any platform adapter can reuse the same lifecycle logic. Platform packages (telegram, slack, etc.) create thin wrappers with their config. Build order: sdk â†’ gateway â†’ pm2 â†’ telegram â†’ cli.
- **[Conventions]**: Library packages using `tsconfig.build.json` (CommonJS for declarations) cannot use `import.meta.url` directly â€” it fails with TS1343. Pass `import.meta.url` as a parameter from the consumer instead, or use `which`/`execSync` for binary discovery.
- **[Architecture]**: Sandbox dependencies install to `/opt/sandbox-deps/` (container-local filesystem), NOT the bind-mounted `node_modules/`. This prevents cross-platform binary conflicts â€” the host (macOS) keeps its darwin binaries, the sandbox (Linux) gets linux binaries separately. PATH and NODE_PATH are set via `buildSandboxEnvWrapper()` in both the shell (`handleShell`) and agent runners (`SandboxedClaudeRunner`/`SandboxedCodexRunner`). The constant `SANDBOX_DEPS_DIR` is in `packages/cli/src/core/sandbox.ts`.
- **[Debugging]**: Biome's Node.js wrapper (`@biomejs/biome/bin/biome`) silently swallows SIGILL crashes: when `spawnSync` returns `status: null, signal: 'SIGILL'`, the wrapper sets `process.exitCode = null` (â†’ exit 0) and never reports the error. Always check `result.signal` when diagnosing silent biome failures. `biome --version` may work while `biome lint` crashes because `--version` uses a simpler code path.
- **[Architecture]**: Bun's module resolution uses `.bun/node_modules/` (a flat symlink directory inside `node_modules/.bun/`) for cross-package resolution. Node.js's `require.resolve` works from within the `.bun/` cache context (via realpath) but NOT from the top-level `node_modules/@scope/` symlinks. Platform-specific optional deps (biome, esbuild) are only symlinked in `.bun/node_modules/`, not at the top level.
